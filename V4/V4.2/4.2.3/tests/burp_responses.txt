## Response Pattern Summary

### Secure (Port 8000) - Always Consistent Pattern

Forbidden Headers Present:
- HTTP Status: 400 Bad Request
- status: "FAIL"
- Contains: forbidden_headers_found array
- Includes: RFC reference

Valid Headers Only:
- HTTP Status: 200 OK
- status: "PASS"
- forbidden_headers_found: [] (empty)

### Vulnerable (Port 8001) - Always Consistent Pattern

Any Request:
- HTTP Status: 200 OK
- status: "ACCEPTED"
- compliance: "FAIL"
- Contains: all_received_headers dict
- Shows dangerous headers but accepts them

---

## Key Differences to Observe in Burp

### Comparison Using Burp Comparer

1. **Status Code Difference:**
   - Secure: 400 for violations
   - Vulnerable: 200 for everything

2. **Response Body Difference:**
   - Secure: Lists forbidden headers found, explains why
   - Vulnerable: Lists all headers received, accepts violation

3. **Message Clarity:**
   - Secure: "FAIL - must not contain connection-specific headers"
   - Vulnerable: "ACCEPTED - without validation - VULNERABLE"

4. **asvs_control Field:**
   - Both include "4.2.3"
   - Secure response indicates compliance success
   - Vulnerable response indicates compliance failure

## Notes for Burp Testing

1. **Status Code Analysis:**
   - Count 400s on secure (should equal number of violations tested)
   - Count 200s on vulnerable (should be same as total requests)

2. **Response Content Analysis:**
   - Use Text search for "FAIL" on secure
   - Use Text search for "ACCEPTED" on vulnerable

3. **Request-Response Pairing:**
   - Save side-by-side comparisons
   - Document exact differences

4. **Volume Testing:**
   - Intruder can send 6x6 = 36 combinations
   - Secure: All should be 400
   - Vulnerable: All should be 200
