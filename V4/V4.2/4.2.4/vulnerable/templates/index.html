VULNERABLE IMPLEMENTATION

Missing CRLF Header Validation

ASVS 4.2.4 · V4.2 HTTP Message Structure Validation · Level 3

This application intentionally omits CRLF header validation. It accepts header values containing CR (\r), LF (\n), and CRLF (\r\n) sequences, demonstrating the risks and what a compliant application must do instead.

⚠️
Educational Notice: Modern HTTP/2 servers (Hypercorn, h2) prevent raw CRLF injection at the wire/framing level. This lab demonstrates the application-layer vulnerability — missing validation logic that must exist per ASVS 4.2.4 regardless of the transport protocol.
❌ ASVS 4.2.4 FAIL
🔓 Validation MISSING
💀 Risk Header Inj.
ASVS Requirement — NOT MET

4.2.4 L3 FAIL

Verify that the application only accepts HTTP/2 and HTTP/3 requests where the header fields and values do not contain any CR (\r), LF (\n), or CRLF (\r\n) sequences, to prevent header injection attacks.

Why this fails: The application performs no validation of header names or values. Any CR, LF, or CRLF in user-supplied data passes through unchecked and may be used in response headers, log entries, or forwarded to downstream systems.

    💀 CRLF Injection Attack Vectors
  

  

    

      
Header Injection

      

        Attacker injects \r\nSet-Cookie: session=attacker into a header value.
        The server adds an attacker-controlled cookie to the response, potentially hijacking
        sessions of other users who receive the poisoned response from a cache.
      

      

        Payload:
        value\r\nSet-Cookie: hijacked=true; Path=/
      

    

    

      
HTTP Response Splitting

      

        Double CRLF (\r\n\r\n) terminates the headers section, allowing the
        attacker to inject a complete second HTTP response. Reverse proxies and CDNs may
        cache this injected response and serve malicious content to all users.
      

      

        Payload:
        value\r\n\r\n<html>Fake Page</html>
      

    

    

      
Log Injection

      

        When server logs record header values, injecting \n allows an attacker
        to write arbitrary lines to the log file. This can hide malicious activity or
        frame innocent users with forged authentication entries.
      

      

        Payload:
        Mozilla\n[INFO] ADMIN logged in from 192.168.1.1
      

    

    

      
Cache Poisoning

      

        Combined with response splitting, CRLF injection can force a shared cache
        (Varnish, Nginx, CDN) to store and serve the attacker's injected response.
        All users requesting the same URL then receive the malicious content.
      

      

        Impact: Mass compromise via shared cache poisoning
      

    

  







  

    🔍 What's Missing in This Vulnerable App
  

  

    

      
The Missing Validation (Python)

      

        The vulnerable application has no before_request hook
        and no validation in any endpoint. User-supplied header data is processed and
        potentially reflected without any CR/LF check.
      

      

        

          ❌
          No before_request hook to validate incoming headers
        

        

          ❌
          No regex check for [\r\n] in header names or values
        

        

          ❌
          No sanitization before logging header values
        

        

          ❌
          No rejection of requests with malformed headers
        

      

      

        Fix: Add the same before_request validation
        as in the secure app. See secure/app.py for the complete implementation.
      

    

    

      
vulnerable/app.py — Missing Validation

      
# VULNERABLE: No before_request hook
# VULNERABLE: No validate_headers() function
# VULNERABLE: Headers processed without checking CR/LF

@app.route("/api/validate-header", methods=["POST"])
def api_validate_header():
    data = request.get_json()
    header_name = data.get("header_name")
    header_value = data.get("header_value")

    # ❌ MISSING: validate_headers({header_name: header_value})
    # ❌ MISSING: if violations: return 400

    # Value used directly — CR/LF sequences pass through
    resp.headers[header_name] = header_value  # DANGER!
    return resp
